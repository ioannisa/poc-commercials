# Reports (JasperReports 7)

PDF reports are generated by JasperReports from visual templates (JRXML).
The system is **generic**: one engine, one wire format, one server endpoint,
and one `ReportService` serve every report. Adding a report adds exactly two
things (see "Adding a new report" below).

## Architecture

```
:reportcore                                  :shared
├─ commonMain (jvm/js/wasmJs)                ├─ commonMain (all platforms)
│  └─ ReportRequest        <- wire envelope  │  ├─ ReportPayload        <- platform-neutral request
└─ jvmMain                                   │  ├─ ProgramFlowReport.kt <- payload builder (per report)
   ├─ ReportEngine         <- generic engine │  ├─ ReportService        <- expect, generic API
   └─ resources/reports/                     │  └─ ui/ReportToolbar
      └─ ProgramFlowReport.jrxml             ├─ reportsMain (jvm+js+wasmJs)
                                             │  └─ ReportWire.kt        <- payload -> wire adapter
:server                                      ├─ jvmMain    ReportService: fills in-process via ReportEngine
└─ POST /api/reports/{reportId}              ├─ webMain    ReportService: POSTs wire request to :server
   (uses ReportEngine + ReportRequest)       └─ android/iosMain  ReportService: "not available"
```

- **Desktop (JVM)** generates in-process; **browsers** POST to the Ktor
  server; both use the *same* engine, templates, and payload builders, so
  output is identical everywhere. **Android/iOS** report "not available".
- There are **no per-report DTOs or bean classes**. The JRXML template is the
  data contract: it declares every parameter and field with a name and Java
  class, and `ReportEngine` coerces the incoming JSON values to those classes
  at fill time. Unknown parameter/field names fail fast with a clear error.
- Compiled templates are cached per report id for the process lifetime.
  Templates are classpath resources baked in at build time, so the cache can
  never go stale: changing a template means rebuilding, which restarts the
  process. Report *data* is never cached.

## Adding a new report

1. **Template** — design in Jaspersoft Studio (JasperReports 7 format) and
   save as `reportcore/src/jvmMain/resources/reports/{ReportId}.jrxml`.
   The id must match `[A-Za-z0-9_-]+`. Use `DejaVu Sans` for Greek text.
2. **Payload builder** — one function in `:shared` commonMain that maps your
   domain data to the template's parameter and field names. Copy the pattern
   from `ProgramFlowReport.kt`:

```kotlin
object InvoiceReport { const val ID = "InvoiceReport" }

fun InvoiceData.toReportPayload(): ReportPayload = ReportPayload(
    reportId = InvoiceReport.ID,
    parameters = buildJsonObject { put("CUSTOMER", customerName) },
    rows = lines.map { line ->
        buildJsonObject {
            put("description", line.description)
            put("amount", line.amount)
        }
    },
)
```

That's it - no engine, route, client, or service changes. A **ViewModel** then
constructor-injects the service (Koin binds it per platform) and owns the
outcome; composables only report the click:

```kotlin
class InvoiceViewModel(private val reportService: ReportService) : BaseGlobalViewModel() {
    private fun export(data: InvoiceReportData, number: String) = viewModelScope.launch {
        when (val result = reportService.exportToPdf(data.toReportPayload(), "Invoice_$number.pdf")) {
            is ReportResult.Success   -> showSnackbar(/* ... */)   // or preview / print
            is ReportResult.Error     -> showSnackbar(UiText.Dynamic(result.message))
            ReportResult.Cancelled    -> showSnackbar(/* ... */)
        }
    }
}
```

`ui/ReportToolbar` is the matching **stateless** component: three buttons plus
`busy`/`available` flags, and it resolves nothing from a container.

## Server API

```
POST /api/reports/{reportId}          -> PDF (attachment)
POST /api/reports/{reportId}/preview  -> PDF (inline)
POST /api/reports/batch               -> many reports as ONE PDF, in order
POST /api/reports/batch/preview       -> same, inline
GET  /api/reports/status              -> health check
```

Body is the serialized `ReportRequest` (`ReportBatchRequest` for the batch
endpoints - note "batch" is therefore reserved as a report id). Errors: `400`
for unknown parameter/field names, an empty batch, or a path/body id mismatch,
`404` for an unknown report id, `500` otherwise.

`ReportService` mirrors this: its methods take `List<ReportPayload>` (with
single-payload convenience overloads), so "print the whole month" is just the
list of that month's daily payloads - see `TimetableViewModel.runMonthReport`.

## Value typing

JSON values are coerced to the class the template declares:
`String`, `Integer`, `Long`, `Short`, `Byte`, `Double`, `Float`, `Boolean`,
`BigDecimal`, `BigInteger` (and `Object`, best-effort). Dates and anything
else should be **pre-formatted strings** built in the payload builder - that
keeps locale/format decisions in one place, in common code.

## Deployment notes

- **Fonts**: the `jasperreports-fonts` extension (DejaVu, Greek support) is
  registered via `jasperreports_extension.properties`; the server fat jar
  merges these files - see the shadowJar config in `server/build.gradle.kts`.
- **Desktop packaging**: ProGuard is disabled for release distributions
  (JasperReports resolves renderers reflectively) - see
  `desktopApp/build.gradle.kts`.
